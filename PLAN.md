## Problem Statement

A personal Mac app that unifies daily office work — LLM chats, files, terminal, Teams,
Slack, email — into one workspace, so the user stops shifting between separate windows
for each of these. Chat sits in the center as the primary "powerhouse" surface; smaller
panels around it hold the rest. Explicitly NOT a developer tool (contrast with Conductor,
which this borrows spatial layout ideas from but does not extend or depend on) — this is
a general daily-work command center for one person.

V1 intentionally ships only a subset of this full vision — the spatial shell plus a
single LLM provider — with comms integrations and multi-provider support sequenced in
afterward (see Next Steps). The sections below distinguish what V1 must hold from what's
an end-state goal that shapes architecture but isn't required to ship first.

## What Makes This Cool

The spatial thesis: one screen, one central focus (chat), tabbed peripheral panels for
everything else, zero window-switching. "A one-stop look at everything I work on daily."
The delight isn't any single integration — it's never having to context-switch across
apps to get through a workday.

## Constraints

**V1 (must hold now):**

- Must look and feel genuinely Mac-native — not a themed web app.
- Left sidebar has two SEPARATE, independent sections (not nested in each other):
  - **Projects** — top-level containers of work, each with its own chats/context
    (like Conductor's repo→workspace grouping, or Claude/ChatGPT's Projects).
  - **Agents** — persistent AI personas you converse with, reusable across every
    project, not scoped to a single one. For V1, an Agent "job" is a synchronous,
    persona-flavored chat thread (a saved system prompt + conversation) — not an
    autonomous background worker with its own tool access. Autonomous execution is
    out of scope until comms integrations (Next Step 8) give an Agent something
    real to act on. Agent *persona definitions* are shared across Projects; each
    Project's *conversation history* with an Agent is scoped to that Project —
    opening the same Agent from a different Project starts a fresh thread, not a
    continued one.
- Center pane is the dominant surface: chat, which can also flip to previewing a
  selected file inline.
- Top-right: a Mac file browser panel.
- Bottom-right: an embedded terminal panel.
- Single LLM provider to start: **Claude via the Claude Agent SDK / Claude Code CLI
  sidecar**, authenticated with the user's existing Claude Code subscription
  (`claude setup-token` / existing login — Claude Code 2.1.212 is already installed
  and authenticated on this machine). This is a supported Anthropic surface, not the
  gray-zone token-capture approach: it satisfies the "use my subscription, not
  metered API billing" goal on day one for Claude. The app spawns the CLI (or a
  small Node sidecar embedding the Agent SDK) as a local subprocess — the same
  architecture Conductor uses. No API key and no Keychain credential are needed for
  V1; the CLI manages its own auth. *(Decision: eng-review Issue 1, option 1A.)*
- Local persistence: chat/Project history stored locally via **SwiftData** (Apple's
  current built-in for new SwiftUI apps — less boilerplate than CoreData, automatic
  lightweight migration, no reason to reach for raw SQLite absent an
  Objective-C-sharing or pre-iOS17 constraint, neither of which applies here); any
  future captured secrets (session tokens, API keys) go in macOS Keychain, never in
  plaintext files.
- Failure-state handling is in scope for V1, not an afterthought: `ModelProvider`
  network errors, invalid keys, and rate limits must surface a visible error in the
  chat pane (not fail silently); a crashed or exited shell in the terminal panel must
  show a clear "process ended" state rather than a frozen pane; permission-denied or
  missing-file cases in the file browser must show an inline message, not silently
  omit the row.
- Sandboxing: V1 ships unsandboxed, using ad-hoc/local Xcode signing for personal use
  (consistent with Premise 4 and the Distribution Plan — no distribution pipeline or
  paid Developer ID enrollment needed) rather than App Sandbox + entitlements, since
  Full Disk Access and a working pty terminal are hard to reconcile with the sandbox's
  file-access model. Revisit signing/sandboxing entirely if this ever needs App Store
  distribution.

**End-state goals (inform architecture direction, not required to ship V1):**

- Multi-provider LLM chat: Claude, GPT, and Grok, with model + effort selection in the
  UI. The V1 provider's own effort levels should be nailed down now (Next Step 3) —
  what's actually unresolved is whether those values map 1:1 across providers once
  GPT/Grok are added, or need a per-provider translation layer; see Open Questions.
- Track usage and daily limits per provider/model — intended as an advisory display of
  consumption, not an enforced hard block. Ships incrementally per provider, starting
  with the V1 provider (simpler to instrument with one provider than with three, not
  harder) rather than waiting for every provider to exist; see Next Steps.
- Authenticate using the user's own subscriptions locally instead of metered API
  billing. **Achieved for Claude in V1** via the Agent SDK sidecar (see V1
  constraints). Remains an open end-state goal only for GPT and Grok, where no
  equivalent supported local-subscription surface is known — see Premise 2 and
  Open Questions for the residual risk on those legs.
- Notifications surface: agents needing a response, priority emails, Teams messages —
  depends on comms integrations (Next Step 8).

## Premises

1. The spatial *layout concept* (sidebar + central chat + file/terminal panels) is
   validated UX — Conductor and Raycast both prove the pattern is usable and well-liked.
   That's a claim about the interaction design, not about implementation effort: a
   from-scratch SwiftUI build still carries real secondary risk in AppKit interop for
   the embedded terminal (pty handling, focus/keyboard routing between panes) and the
   custom pane layout. Low risk relative to the auth bet below — not zero risk in
   absolute terms.
2. *(Revised at eng review.)* The subscription-auth bet split into two very different
   halves. **Claude: solved via a supported surface** — the Claude Agent SDK / Claude
   Code CLI uses the user's existing subscription login (`claude setup-token`), no
   gray zone, no fragility beyond normal SDK versioning. **GPT/Grok: still the
   original high-risk bet** — no known equivalent supported local-subscription
   surface; the candidate mechanism there remains capturing the session/auth token
   their own web/desktop clients use, which likely sits in a gray zone against
   consumer ToS and can silently break when the provider changes their web app.
   Time-box any GPT/Grok spike to 3-5 days per provider before deciding whether to
   keep pursuing it or settle for API billing on those legs.
3. "Agents I assign non-coding jobs to" (email triage, Teams watching) requires real
   tool-access integrations per surface (Gmail/Graph/Slack APIs) — each is its own
   small project, separate from the chat UI itself. This is why V1's Agents ship as
   synchronous persona-chat only (see Constraints) rather than autonomous workers.
4. The V1 wedge (sidebar + folder tree + central chat + file browser + terminal, single
   LLM provider via official API) needs no distribution plan — personal use only, no
   App Store or installer pipeline required yet. This is also why V1 can skip App
   Sandbox entitlements (see Constraints).
5. Nothing on the market combines all four pillars — native persistent Mac shell,
   multi-provider local-auth chat, file/terminal panels, and unified comms — in one
   single-user product. Landscape research (OpenAI Workspace Agents, Raycast, and the
   unified-inbox category: Universal Inbox, Notico, this+that, Missive) confirms the
   closest players each cover only part of this, split across enterprise/cloud tools,
   Mac launchers, and inbox aggregators. The gap exists because none of them have a
   commercial reason to build for exactly one person's machine — not because it's
   technically unusually hard.

All five premises were presented to the user; none were disputed. The user instead
added detail (the Projects/Agents sidebar split) rather than pushing back, which is
being treated as agreement.

## Approaches Considered

### Approach A: Fast Wedge (Tauri/Electron shell, API-key auth)
Web-tech shell (Tauri preferred over Electron for a lighter, closer-to-native feel)
using off-the-shelf components — xterm.js for the terminal, a file-tree library,
Monaco/CodeMirror for preview — with one LLM wired up via its official API key.
Effort: S/M. Risk: Low.
Pros: fastest realistic path to something running this week; huge ecosystem of
ready-made components to wire together.
Cons: even Tauri's native webview won't fully deliver the "looks Mac native" bar set
in the constraints; API-key billing is the metered-cost model the user explicitly
wants to avoid, and this approach doesn't address that at all.

### Approach B: Ideal Architecture (Native SwiftUI/AppKit, subscription-auth first)
Full native app — SwiftUI sidebar, SwiftTerm-based terminal pane, FileManager+QuickLook
for browsing — with real subscription-based auth solved for at least Claude before
anything else ships.
Effort: L/XL. Risk: Med-High.
Pros: zero-compromise native look and feel with no retrofit risk later; once built,
it's the real product, not a placeholder.
Cons: no usable daily tool until the hardest, most uncertain part (subscription auth)
is solved, which could stall the whole project for weeks; that research could hit a
wall (ToS enforcement, device-binding) with nothing to show for the time spent.

### Approach C: Native Shell Now, Provider Behind an Abstraction (chosen)
*(Text revised at eng review — D5.8 — to match the decisions that superseded the
original API-key framing.)*
Build the real native SwiftUI shell immediately — Projects list, Agents list, central
chat/preview pane, file browser, terminal — with LLM calls behind a small
`ModelProvider` interface. The V1 provider drives the installed Claude Code CLI
using the user's existing subscription (see Constraints and Next Step 2); API-key
providers for GPT/Grok come later behind the same interface.
Effort: M. Risk: Low/Med.
Pros: delivers the native feel and a daily-usable app with subscription auth working
from day one for Claude; isolates the remaining auth unknown (GPT/Grok subscription
surfaces) so a dead end there costs research time, not the project; matches the
user's own stated wedge almost exactly. Realistic timeline (D5.7): **2-4 weeks**,
not the original 1-2 — the 4-pane layout, provider abstraction, pty terminal, and
chat markdown rendering are real surface area, the builder's Swift/AppKit fluency
is unestablished, and AI assistance is measurably weaker on niche AppKit interop
than on web stacks.
Cons: requires upfront interface design instead of hardcoding one integration; the
app shares the user's Claude subscription rate window with their coding work (see
Step 2 usage note). Residual risk: the swap-in premise assumes future providers'
surfaces expose comparable capabilities (effort control, session semantics) — if
GPT/Grok don't, the `ModelProvider` abstraction may need reshaping when they land.

## Recommended Approach

**Approach C.** It's the only option that honors both the hard V1 constraint
(genuinely native Mac feel) and the end-state goal (eventually using real
subscriptions instead of API billing) — without letting the second one block the
first. The shell work (relatively low-risk per Premise 1, though not risk-free —
see the AppKit/pty caveats there) happens regardless of which auth method wins, and
quarantining the genuinely uncertain part (Premise 2) means a dead end there costs
research time, not the whole project.

## Open Questions

- Whether Conductor (or any comparable tool) exposes a plugin/extension surface worth
  building on, versus this being a fully separate native app — not established either
  way; worth checking before any decision to integrate rather than build fresh.
- Whether Slack should follow Teams or precede it (Next Step 8 sequences email first;
  the order of the other two is still unset — likely whichever causes more daily
  interruption today).
- Whether local subscription-based auth is viable long-term for **GPT and Grok**
  (the Claude leg is resolved — see Premise 2 revision), given ToS and fragility
  risk — needs the time-boxed research spike before real commitment, independent of
  and not blocking the shell build.
- Whether the V1 allowlist-based "priority" detection (Next Step 8) is good enough once
  real usage shows what actually gets missed, or whether it needs to grow into
  something smarter (e.g. classification) — deferred until there's real usage to learn
  from.
- Whether effort levels (defined concretely for Claude in Next Step 3) map 1:1 across
  Claude/GPT/Grok, or need a per-provider translation layer — unresolved until
  provider #2 is added (Next Step 9).

## Success Criteria

V1 is done when the user can open the app each morning and, without switching to a
separate Terminal.app, Finder window, or chat app tab: continue a chat with one LLM
provider in the center pane, browse and preview Mac files top-right, and run terminal
commands bottom-right, switching between Projects and Agents from two independent
sidebar lists. Concretely: used as the daily driver for those tasks for a full work
week without reverting to the old separate tools for them.

## Distribution Plan

None needed for V1 — this is for personal use only, so no App Store listing, no
installer/update pipeline, no code-signing pipeline required yet. Revisit only if the
project ever moves beyond single-user use.

One hard constraint if that ever happens: Anthropic's Agent SDK docs state that
third-party developers may not offer claude.ai login or subscription rate limits to
*their* users without prior approval. Driving your own installed, logged-in Claude
Code from your own personal tool is a different situation — but distributing this
app to others would require rethinking the auth model (API keys or an approved
arrangement), not just adding an installer.

## Next Steps

**Build order (eng-review D5.5): 1 → 6 → 2 → 3 → 4 → 5.** The Projects/Agents data
model and sidebar (Step 6) come *before* the provider and chat wiring (Steps 2-3),
because per-Project working directories and threads are prerequisites for
conversation continuity — the chat built first would have nowhere to live. Files (4)
and terminal (5) follow the chat path; the app is daily-usable after Step 3. Step
numbering below is unchanged; this is execution order.

1. Scaffold the native SwiftUI Mac app project with the 4-pane layout: left sidebar
   split into Projects and Agents sections, center chat/preview pane, top-right file
   browser, bottom-right terminal. All color usage goes through semantic/adaptive
   color definitions from the first view onward (design review D7) — no raw hex in
   views — so Light and Dark mode both work without retrofit; includes a short
   Light-value derivation pass since the wireframes only established the Dark set.
2. Build the `ModelProvider` abstraction; implement the first concrete provider by
   driving the **installed `claude` CLI as one long-running process per conversation**
   (`--input-format stream-json --output-format stream-json`), with spawn-per-turn
   (`-p` + `--resume`) as the fallback if the persistent mode misbehaves (eng-review
   decision 2A as revised by outside-voice D5.2 — a persistent process eliminates
   per-turn cold start: config load, MCP init, and session deserialization that
   would otherwise add multi-second first-token latency to every message). Not a
   Node sidecar; that remains a later swap behind the same abstraction.
   **First task of this step, before any UI work (D5.1): prototype one real
   conversation through this pipe and judge the chat quality.** If the scoped chat
   feels worse than a claude.ai browser tab, the provider configuration gets fixed
   first — a native shell around a bad chat fails V1's own success criterion.
   Invocation surface **verified against code.claude.com docs (2026-07-18)**:
   - Non-interactive: `claude -p "<prompt>"`; streaming via
     `--output-format stream-json --verbose --include-partial-messages`. Text deltas
     arrive as `stream_event` lines where `event.delta.type == "text_delta"`; the
     final line is a `result` message with the full text, cost, and session metadata.
   - Continuity: capture `session_id` from the `system/init` event (or
     `--output-format json` → `.session_id`), then `--resume <session_id>` for
     follow-up turns. **Session lookup is scoped to the working directory** — spawn
     the CLI with `cwd` set per-Project (a stable directory per Project) so resume
     works reliably. **Resume-failure fallback (D5.3):** Claude Code garbage-collects
     transcripts (`cleanupPeriodDays`, ~30-day default), so a Project untouched for a
     month would fail `--resume`. Two-part mitigation: raise retention via
     `--settings`, AND on resume failure rehydrate a fresh session by replaying the
     thread history from SwiftData (SwiftData is the durable store; CLI transcripts
     are a cache).
   - **Chat defaults (D5.1 — requirements, not options):** full `--system-prompt`
     *replace* (never `--append-system-prompt` — appending a persona to the
     coding-agent base prompt creates an identity conflict the base prompt wins);
     tools disabled by default for plain chat; and scope the harness so the user's
     global config can't contaminate conversations: `--setting-sources` restriction,
     `--strict-mcp-config`, `--disable-slash-commands`. Without these, this machine's
     global CLAUDE.md routing rules and ~100 gbrain MCP tool schemas would load into
     every chat. Graduate to `--allowedTools` grants when Agents later gain real
     capabilities.
   - Model/effort: `--model` flag; `/model` and `/effort` also work as in-prompt
     commands in `-p` mode (v2.1.205+).
   - **Do NOT use `--bare`**: bare mode skips OAuth/keychain reads and requires an
     API key — it breaks subscription auth. Spawn without `--bare` so the CLI uses
     the existing login; accept that it loads `~/.claude` config, or scope loading
     with settings flags.
   - Errors: `system/api_retry` events carry typed categories
     (`authentication_failed`, `billing_error`, `rate_limit`, `overloaded`, …) —
     map these to the visible chat-pane error states required by Constraints.
     **One banner shape, dynamic text/CTA per category (design review D5):**
     `authentication_failed` → "You're logged out of Claude Code." / "Open
     Terminal to re-auth" (no Retry — retry can't fix a logged-out session);
     `billing_error` → "Billing issue with your Claude subscription." /
     "Details"; `rate_limit` → "Claude hit its usage window (resets {time})."
     / "Retry" (matches existing mockup copy); `overloaded` → "Claude is
     temporarily overloaded." / "Retry"; CLI process died (not a typed
     category — subprocess exited) → "Lost connection to Claude." / "Restart".
     Every category keeps a partial-reply preserved + tagged "[partial —
     kept]" per the existing mockup.
   - Usage (D5.4): the binding constraint on subscription auth is **rate-window
     consumption, not dollars** — `total_cost_usd` reports notional cost the user
     doesn't pay. Track turns/tokens and treat `rate_limit` error events as ground
     truth for Step 10's display. Acknowledge the tradeoff plainly: this app shares
     the same subscription quota as the user's actual Claude Code usage; heavy chat
     here can consume the window their coding work needs.
   - Hardening: feature-detect via the `system/init` event's `capabilities` array
     (v2.1.205+) rather than comparing version strings; confine all CLI coupling to
     the single provider file so version drift breaks one file, not the app; drain
     stderr continuously on the subprocess (an undrained stderr pipe deadlocks the
     child once the buffer fills). No `CredentialProvider` protocol yet — the
   CLI manages its own auth; add that protocol when the first API-key-based provider
   (GPT/Grok, Step 9) actually lands.
   - **Background process lifecycle across Project switches (design review Pass 7
     #15):** an LRU-capped number of Projects (start with 3) keep a persistent CLI
     process alive in the background when unfocused; beyond the cap, the oldest
     falls back to spawn-per-turn (the existing D5.2 fallback path). A background
     process finishing mid-reply is exactly what the sidebar's unread-activity dot
     (see Visual System) surfaces — switching Projects mid-stream is never
     destructive, matching the User Journey table's existing "nothing lost" beat.

   Chat data flow (V1):

   ```
   ┌────────────────────────── SwiftUI app ──────────────────────────┐
   │                                                                 │
   │  Sidebar                 Center pane              Right panels  │
   │  ┌──────────┐            ┌─────────────┐          ┌───────────┐ │
   │  │ Projects │─select────▶│  Chat view  │          │ FileTree  │ │
   │  ├──────────┤            │  (or file   │◀─preview─│ (top)     │ │
   │  │ Agents   │─select────▶│   preview)  │          ├───────────┤ │
   │  └──────────┘            └──────┬──────┘          │ SwiftTerm │ │
   │        │                        │ send/stream     │ (bottom)  │ │
   │        │                 ┌──────▼──────┐          └─────┬─────┘ │
   │        │                 │ModelProvider│                │ pty   │
   │        │                 │  protocol   │                ▼       │
   │        │                 └──────┬──────┘          /bin/zsh      │
   │        │                        │                               │
   │  ┌─────▼────────┐      ┌───────▼─────────┐                     │
   │  │  SwiftData   │◀─────│ ClaudeCLI       │                     │
   │  │ (Projects,   │ save │ Provider        │                     │
   │  │  Agents,     │      └───────┬─────────┘                     │
   │  │  Threads,    │              │ persistent process per convo  │
   │  │  Messages)   │              ▼ (stream-JSON stdio, D5.2)     │
   │  └──────────────┘      `claude` CLI (headless)                 │
   ```

3. Wire the central chat pane to the `ModelProvider`, with a model + effort picker in
   the composer. Effort values for V1 map to Claude Code's own effort settings —
   confirm the concrete surfaced levels during Step 2's doc check; only the
   cross-provider mapping is deferred (see Open Questions). Includes **streaming
   markdown rendering** for chat output (D5.7 — a real, sized work item, not an
   afterthought: pick a Swift markdown renderer that tolerates incremental text).
   **Composer usage meter (design review Pass 7 #18):** the "Window: X% used"
   pill in wf-1 is a mockup-only preview of Step 10 — omit it from the V1
   composer entirely (not even a placeholder) until Step 10 actually lands;
   this is a documented, intentional deviation from wf-1, not an oversight.
   **Single thread per Project for V1 (Pass 7 #17):** the data model allows
   multiple Threads, but V1 ships exactly one active thread per Project — the
   tab bar's "+" opens file-preview tabs only, never a second chat thread;
   multi-thread-per-Project is explicitly deferred, not silently half-built.
4. Implement the file browser panel (FileManager-based tree/list, simple preview) and
   wire file selection to flip the center pane into preview mode. **Lazy per-directory
   loading** (eng-review decision 4A): enumerate only a directory's immediate children,
   on expand, off the main thread — never recursively walk the tree up front. A real
   home directory holds 100k+ files; eager enumeration would hang the UI on first open.
   **"Discuss in chat" (⌘⏎, design review Pass 6 #3):** switches the center pane to
   the Chat tab and inserts a lightweight reference to the open file (path + line
   range if a selection exists) into the composer as plain text the user can edit
   before sending — not a silent auto-send, not a rich attachment object.
5. Embed a terminal panel using SwiftTerm (proven in shipped apps — Secure Shellfish,
   CodeEdit). Implementation notes from research: its macOS front-end is an AppKit
   `NSView` (`TerminalView`), so it needs an `NSViewRepresentable` wrapper, not a
   native SwiftUI view; data-feed calls must be dispatched to the main thread; PTY
   dimensions need explicit handling when the shell swaps to the alternate screen
   buffer (vim, less, etc.); a crashed/exited shell must surface a visible
   "process ended" state per the Constraints failure-handling requirement.
   **Keyboard passthrough contract (design review Pass 6 #1-2 — resolves the
   focus/keyboard-routing risk Premise 1 already flagged):** subclass/wrap
   `TerminalView` to override `performKeyEquivalent:`, returning false (pass
   through) for exactly ⌘1-⌘4, and forwarding every other key — including Esc —
   to the terminal untouched. Esc-closes-file-preview is bound only to the
   file-preview view's own first-responder state and must never fire while the
   terminal is focused. Add to this step's verify: "⌘1-⌘4 switch panes from
   inside a running shell; open vim, confirm Esc stays inside vim and never
   flips the center pane."
6. Build the Projects list (create/switch, each with its own chat history/context) and
   the Agents list — create/converse with a saved persona (system prompt + synchronous
   conversation only, no autonomous tool access yet, per Constraints) — as two
   independent sidebar sections, confirmed explicitly not nested. Projects and Agents
   remain two clearly separate, separately-headed sections on screen (per the approved
   wireframe) with independent data sources — the only thing shared between them is a
   single small `SidebarRowView` used to render one row's icon/label/status dot in
   both sections, purely to avoid duplicating that rendering code. This is a
   code-reuse detail only; it does not merge or blur the two sections visually or
   architecturally. **Archive (design review D3):** Project gets an `archived: Bool`
   field; sidebar shows a plain unfiltered list by default with a per-row Archive
   action and an active/archived filter toggle — same pattern as Claude's/ChatGPT's
   own Projects lists. Scoped to Projects only; Agents don't get this in V1.
   **Launch behavior (design review D6):** Project gets a `lastActiveAt` timestamp,
   updated on selection; app launch auto-selects the most recent Project and
   resumes its most recent thread (via the D5.3 session-resume path) rather than
   showing a picker — first-run-ever (no Projects exist) still shows the wf-2
   greeting screen, unaffected by this.
   **Working-folder binding (design review Pass 7 #14):** Project creation offers
   a folder picker (choose any existing directory; Project name defaults to the
   folder name, editable) with an "or start empty" fallback that creates a fresh
   folder under `~/Projects/<name>`. Both real cases the wireframes already imply
   — an existing folder like Tax Prep 2026, and starting something new — are one
   creation flow, not two.
   **"Quick chat" entry point (design review Pass 7 #13):** the first-run "or
   start a quick chat ⏎" link and any Agent conversation started before a Project
   exists both auto-create a real (initially hidden/unlabeled-in-sidebar) "Quick
   Chat" Project behind the scenes — not a new ephemeral/Project-less code path.
   The whole architecture (CLI cwd, session-resume, threads) is already
   Project-scoped; this keeps it that way instead of adding a second identity
   model for zero real benefit. The user can rename/promote it like any Project
   once it has real content.

### Information Architecture

```
PRIMARY   → Center pane: active chat thread, or flipped to file preview
SECONDARY → Left sidebar: Projects list (top) / Agents list (bottom) — nav + identity
TERTIARY  → Right column: Files (top) / Terminal (bottom) — supporting context

First screen, in order of attention:
  1. Center pane content (chat history or file preview) — default keyboard
     focus (⌘2), where the eye lands first
  2. Sidebar selection state — which Project/Agent is active right now
  3. Right-column headers (Files path, Terminal cwd) — orientation only,
     until the user clicks in
```

Constraint worship — if only 3 things are visible on open: (1) the active
conversation, (2) which Project it belongs to, (3) that Files/Terminal exist
and are scoped to that Project. Everything else (agents list, kbd hints,
window-usage meter) is secondary chrome.

**Sidebar scale (D3):** Projects list shows all projects by default (plain
list, no search box) with a per-project **Archive** action; a filter/toggle
switches the view between active and archived — same pattern as Claude's and
ChatGPT's own Projects lists. Archiving is what keeps the list usable long
after "Tax Prep 2026" is done; the plain list stays simple for the common
case of a handful of active projects. Touches Step 6's data model (needs an
`archived` flag on Project) and the sidebar UI (filter control + archive
action, likely a right-click/hover affordance per row). Scoped to Projects
only — Agents are reusable personas you don't accumulate the same way, so the
same problem is unlikely to hit them in V1; revisit only if it does.

### Responsive & Accessibility

**Keyboard & pane focus — resolves the pty risk Premise 1 already named:**
- ⌘1–⌘4 pane-switching: SwiftTerm's `TerminalView` is subclassed/wrapped to
  pass through exactly these four chords (override `performKeyEquivalent:` to
  return false for them, forward everything else) rather than trusting
  default AppKit dispatch.
- Esc: scoped to the file-preview view only (`.keyboardShortcut(.escape)` bound
  to that view, never window-level) — must NEVER fire while the terminal
  NSView is first responder.
- ⌘2 = unconditional "go to Chat tab," always. ⌘⏎ (from a file preview) = go
  to chat AND insert a reference to the open file into the composer.

**Window behavior:**
- Below a width breakpoint where the 3-column layout no longer comfortably
  fits, the right column (Files/Terminal) collapses to an overlay/drawer —
  same pattern as Mail/Xcode at narrow widths. No hard minimum window size.
- Sidebar/center/right-column dividers are draggable (`NSSplitView`), widths
  persisted per-window.
- Large/external displays: center-pane chat content caps at ~900-1000px
  reading width; everything else uses AppKit's stock frame/position restore.

**Accessibility:**
- Status dots carry a SwiftUI accessibility label/value ("Streaming" / "Idle")
  alongside color.
- Per-row Archive action is focus-visible and keyboard-activatable
  (Space/Enter) or bound to the standard macOS contextual-menu key.
- Interactive elements use stock SwiftUI `List`/`Button` components
  (platform-default hit areas) rather than custom-drawn controls.
